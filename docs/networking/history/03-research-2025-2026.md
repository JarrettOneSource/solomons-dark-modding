**Executive Summary**
- The biggest correction is terminology and API choice: do not build on old Steam “P2P”/`ISteamNetworking`; build on `ISteamNetworkingSockets` plus Steam Datagram Relay where available. Valve now explicitly marks `ISteamNetworking` deprecated and may remove it (https://partner.steamgames.com/doc/features/multiplayer/networking, https://partner.steamgames.com/doc/features/multiplayer/steamdatagramrelay).
- The biggest design correction is authority: your current “local player = self-authoritative pose stream” is too weak. Current practice is server/host authority with client prediction and reconciliation, plus snapshot interpolation for everyone else; strict lockstep-only is still painful in real ARPGs under jitter (https://forum.lastepoch.com/t/official-last-epoch-multiplayer-faq/42036, https://fish-networking.gitbook.io/docs/guides/features/prediction/what-is-client-side-prediction, https://mirror-networking.gitbook.io/docs/manual/components/network-transform/snapshot-interpolation, https://doc.photonengine.com/fusion/current/concepts-and-patterns/network-simulation-loop, https://www.pathofexile.com/forum/view-thread/3887145).
- Your “headless-ish exe first, extracted sim later maybe never” instinct is directionally right. Most comparable games still ship a dedicated server executable/tool rather than a cleanly separated server product (Valheim, Enshrouded, V Rising, Palworld) (https://www.valheim.com/support/a-guide-to-dedicated-servers/, https://enshrouded.zendesk.com/hc/en-us/articles/16051370691485-Dedicated-Server-Installation-on-Steam, https://guides.playvrising.com/hc/en-us/articles/5545817790877-How-to-Host-a-Server, https://docs.palworldgame.com/getting-started/deploy-dedicated-server).
- The thing you are most missing is not movement math, it is session/reconnect/full-state-resync/admin/trust plumbing. Last Epoch’s launch retrospective is basically a warning that party/matcher/service design can sink multiplayer before movement ever matters (https://forum.lastepoch.com/t/1-0-launch-retrospective/69374).
- Public disclosures on transport and tick rate are still sparse. Where studios have not published them, I am not guessing.

**Field Scan**
- `Diablo IV`: clearly server-backed, always-online shared-world/cross-play/cross-progression; Blizzard publicly emphasized server durability testing via Server Slam, but I found no primary-source public disclosure of transport or tick rate (https://news.blizzard.com/en-us/diablo4/23938755/help-us-gear-up-for-launch-during-the-diablo-iv-server-slam, https://news.blizzard.com/en-us/diablo4/23189677/diablo-iv-feature-overview).
- `Path of Exile 2`: public evidence still points to lockstep/predictive-style networking modes; the big reported pain is latency spikes translating directly into freezes/stutter, which is exactly the lockstep failure mode you want to avoid as your only option (https://www.pathofexile.com/forum/view-thread/3887145, https://www.pathofexile.com/forum/view-thread/3831045).
- `Last Epoch`: official messaging is online multiplayer separated from offline to prevent cheating; their FAQ says multiplayer is neither pure predictive nor pure lockstep, and the 1.0 retrospective says the real disasters were matcher/party-service scaling and join timeouts (https://forum.lastepoch.com/t/official-last-epoch-multiplayer-faq/42036, https://forum.lastepoch.com/t/1-0-launch-retrospective/69374).
- `No Rest for the Wicked`: Moon’s 2025-2026 messaging is a strong signal toward persistent shared “realms” rather than throwaway host sessions; that is closer to V Rising/Valheim server-world thinking than classic temporary ARPG join/drop-in (https://norestforthewicked.com/news/being-wicked-together, https://norestforthewicked.com/news/3-2-1-together).
- `V Rising`: persistent-world model, dedicated server tool, private server save transfer, and clear split between official-server moderation and private-server owner authority (https://guides.playvrising.com/hc/en-us/articles/5545817790877-How-to-Host-a-Server, https://guides.playvrising.com/hc/en-us/articles/28560453725725-How-to-Transfer-Save-Files-to-a-New-Host-in-V-Rising-Private-Server, https://guides.playvrising.com/hc/en-us/articles/28561764665117-Ban-Appeals-in-V-Rising-How-to-Submit-an-Appeal).
- `Enshrouded`: separate dedicated server exe plus server browser that lists both dedicated and peer-to-peer sessions; dedicated is a first-class path, not an afterthought (https://enshrouded.zendesk.com/hc/en-us/articles/16051370691485-Dedicated-Server-Installation-on-Steam, https://enshrouded.zendesk.com/hc/en-us/articles/16056312924957-Dedicated-Server-FAQ).
- `Valheim`: dedicated server tool, Steam backend by default, PlayFab backend when crossplay is enabled; adminlist/banlist/permittedlist rather than heavyweight anti-cheat (https://www.valheim.com/support/a-guide-to-dedicated-servers/, https://valheim.com/support/crossplay-faq/).
- `Palworld`: dedicated server app plus admin-password/kick/ban tooling; again, practical server ops over invasive anti-cheat (https://docs.palworldgame.com/getting-started/deploy-dedicated-server, https://docs.palworldgame.com/settings-and-operation/commands/).
- `Lost Ark`: the 2025 NA region merge explicitly kept game worlds in their original server locations to mitigate latency/ping while merging market/social layers. That is a useful reminder that social region and sim placement do not have to be the same thing (https://www.playlostark.com/en-us/news/articles/region-merge-update).
- `Grim Dawn`, `Torchlight Infinite`: I did not find useful 2025-2026 primary-source netcode disclosures worth pretending are stronger than they are.

**Findings**
1. Steam transport has moved decisively away from old raw Steam P2P APIs.
`ISteamNetworking` is now explicitly “older, deprecated” and Valve says to migrate; the current docs say the newest APIs relay through Valve by default, support ordinary UDP, and automatically use SDR for P2P when appropriate (https://partner.steamgames.com/doc/features/multiplayer/networking, https://partner.steamgames.com/doc/features/multiplayer/steamdatagramrelay).
Should this change our plan? `Yes.`
Change decision to: `SteamNetworkingSockets`/SDR as the Steam transport, not “Steam P2P” in the old sense.

2. For a mod framework, do not make Steam-hosted SDR features your only path.
Valve’s docs and the GameNetworkingSockets README are clear that the open-source code does not get you Steam’s relay network/auth/signaling, and hosted dedicated SDR assumes Steamworks app/partner/data-center setup (https://partner.steamgames.com/doc/features/multiplayer/steamdatagramrelay, https://github.com/ValveSoftware/GameNetworkingSockets/blob/master/README.md).
Should this change our plan? `Yes.`
Change decision to: keep Steam transport primary, but require a direct-IP/dedicated fallback that does not depend on privileged Steam partner features. This matters a lot for a DLL mod riding inside someone else’s retail app.

3. “Local player self-authoritative pose stream” is too weak; use input authority plus host/server reconciliation.
Current mainstream tooling and backend docs all converge on this: server-authoritative state, client prediction for responsiveness, snapshot interpolation for remote objects (https://fish-networking.gitbook.io/docs/guides/features/prediction/what-is-client-side-prediction, https://mirror-networking.gitbook.io/docs/manual/components/network-transform/snapshot-interpolation, https://doc.photonengine.com/fusion/current/concepts-and-patterns/network-simulation-loop, https://heroiclabs.com/docs/nakama/concepts/multiplayer/authoritative/). Last Epoch’s public stance is also “responsive, but not pure predictive or pure lockstep,” while PoE2’s public pain shows why strict lockstep-only is brittle in ARPG combat (https://forum.lastepoch.com/t/official-last-epoch-multiplayer-faq/42036, https://www.pathofexile.com/forum/view-thread/3887145).
Should this change our plan? `Yes.`
Change decision to: client sends inputs; authoritative host/dedicated simulates; local client predicts and reconciles; remote players/enemies interpolate snapshots.

4. Keep headless-ish exe as phase 1.
Comparable games still ship a server executable/tool, not a clean-room extracted sim: Valheim, Enshrouded, V Rising, and Palworld all document a separate server binary/toolchain and config/save/admin files (links above).
Should this change our plan? `No.`
Keep current decision, but force a clean `IGameNetDriver`/`ITransport` seam now so you can later replace the fake-headless server without rewriting replication.

5. The packet catalog is underdesigned for what actually breaks in production.
Last Epoch’s retrospective is the cautionary tale: matcher design flaws and party-service checks caused catastrophic failures at load, not exotic movement code (https://forum.lastepoch.com/t/1-0-launch-retrospective/69374). Steam’s lobby docs also explicitly describe a lobby-to-host/server transition, not “just start streaming packets” (https://partner.steamgames.com/doc/features/multiplayer/matchmaking).
Should this change our plan? `Yes.`
Change decision to: define v1 packet families for `hello/version/mod-manifest`, `join/auth`, `clock-sync`, `full-state bootstrap`, `delta snapshot`, `spawn/despawn`, `reliable gameplay event`, `inventory/drop/pickup`, `damage/result`, `boss/world transition`, `disconnect/reconnect`, and `save provenance`.

6. Your dedicated + P2P hybrid should look like Enshrouded more than like a true mesh P2P game.
Enshrouded’s browser can show both dedicated and peer-to-peer sessions, with dedicated favored/sorted separately; Valheim similarly exposes dedicated plus different backends depending on mode (https://enshrouded.zendesk.com/hc/en-us/articles/16056312924957-Dedicated-Server-FAQ, https://www.valheim.com/support/a-guide-to-dedicated-servers/).
Should this change our plan? `Yes, slightly.`
Change decision to: treat hosted game and dedicated game as the same client/server topology with different session discovery, not as two different networking architectures.

7. 10 Hz enemies is probably too low for melee ARPG combat as a default.
Mirror’s docs are a good reminder that snapshot interpolation works well, but only if you keep feeding the buffer; Photon’s docs also reinforce a fixed tick/snapshot model decoupled from render (https://mirror-networking.gitbook.io/docs/manual/components/network-transform/snapshot-interpolation, https://doc.photonengine.com/fusion/current/concepts-and-patterns/network-simulation-loop).
Should this change our plan? `Yes.`
Change decision to: design around 15-20 Hz enemy snapshots minimum for nearby combat-relevant enemies, with velocity + action/state flags + hit react/stagger state. Use 20-30 Hz for player snapshots if you are snapshotting players at all; better is input at 30-60 Hz and render interpolation separately.

8. Use unreliable snapshots and reliable events, not reliable transform streams.
Mirror explicitly warns its `NetworkTransform` sends snapshots on an unreliable channel every send interval; reliable is for events that must land, not for continuously superseded pose data (https://mirror-networking.gitbook.io/docs/manual/components/network-transform/snapshot-interpolation).
Should this change our plan? `No, but it sharpens it.`
Keep current decision, but formalize the split now: unreliable snapshot lanes for movement/state, reliable lanes for casts, drops, joins, confirmations, and progression.

9. Host-authoritative P2P cannot truly solve trusted-host abuse.
Valve’s own anti-cheat docs say authoritative servers are strongly suggested, and if you use P2P, peers must validate each other so the host does not have full control; V Rising, Valheim, and Palworld all lean heavily on server-owner/admin tooling and private-server trust, not magic client anti-cheat for private hosts (https://partner.steamgames.com/doc/features/anticheat, https://guides.playvrising.com/hc/en-us/articles/28561764665117-Ban-Appeals-in-V-Rising-How-to-Submit-an-Appeal, https://www.valheim.com/support/a-guide-to-dedicated-servers/, https://docs.palworldgame.com/settings-and-operation/commands/).
Should this change our plan? `Yes.`
Change decision to: skip EAC/BattleEye/VAC-style ambitions for the mod framework v1. Instead, disable the named-pipe Lua exec server in multiplayer, partition “trusted private host” from “dedicated/secure” worlds, validate hits/loot/xp server-side, and make official progression require dedicated authority if you ever add it.

10. Non-Steam dedicated transport: do not start with QUIC/WebTransport.
MsQuic is real, modern, and now has secure datagrams; it even has newer work like QTIP/XDP-adjacent transport tricks. But it brings TLS/cert/handshake/ops complexity you do not need for a 4-player x86 DLL mod in v1 (https://microsoft.github.io/msquic/msquicdocs/docs/API.html, https://microsoft.github.io/msquic/msquicdocs/docs/QTIP.html). Meanwhile, GameNetworkingSockets is explicitly game-focused, supports reliable/unreliable messages, lanes/channels, ordinary UDP, and has a Steam-aligned API surface; ENet remains the simple small-footprint fallback (https://github.com/ValveSoftware/GameNetworkingSockets/blob/master/README.md, https://github.com/ValveSoftware/GameNetworkingSockets/releases, https://github.com/lsalzman/enet).
Should this change our plan? `Yes.`
Change decision to: do not choose QUIC first. Either use standalone GameNetworkingSockets if you want API symmetry with Steam, or ENet if you want the leanest off-Steam dedicated backend.

11. Matchmaking/session best practice for your shape is still simple.
Steam’s docs still describe lobbies as the grouping/invite layer and then a handoff to the game host/server; they explicitly warn against high-frequency metadata abuse (https://partner.steamgames.com/doc/features/multiplayer/matchmaking). PlayFab and GameLift are now cleaner for server allocation if you outgrow hobby hosting, but they are later-stage infrastructure, not v1 necessities (https://learn.microsoft.com/en-us/gaming/playfab/multiplayer/servers/using-playfab-servers-to-host-games, https://docs.aws.amazon.com/gamelift/latest/developerguide/fleets-creating-anywhere.html).
Should this change our plan? `No, mostly.`
Keep current decision: Steam lobby for party/invites/readiness, then connect to host or dedicated by ticket/address. Add direct IP and Steam favorites/server-browser support for dedicated. Do not invent a custom cloud coordinator unless scale forces it.

12. Surprise: the persistent-world/server-config model is becoming the norm for co-op survival-ARPG-adjacent games.
V Rising, Valheim, Enshrouded, Palworld, and No Rest for the Wicked all point toward “world/server identity matters” rather than disposable quickplay sessions (links above).
Should this change our plan? `Yes, conceptually.`
Change decision to: treat the world/save/server as a first-class replicated object. Add versioning, migration, ownership, backup/restore, and mod-manifest compatibility checks early. That is an inference from the field plus your own modded/Lua-heavy environment, and I think it matters.

**Concrete Recommendation Set**
- Transport: `ISteamNetworkingSockets` + SDR on Steam; keep your `ITransport` abstraction; add a non-Steam dedicated backend later using standalone GameNetworkingSockets if you want API symmetry, or ENet if you want minimum integration cost. Do not build new work on deprecated `ISteamNetworking`.
- Replication paradigm: authoritative host/dedicated simulation with client-side prediction + reconciliation for the locally controlled player, snapshot interpolation for remote players/enemies, reliable event streams for casts/drops/progression. Do not use full-game rollback; do not use strict lockstep as the only mode.
- Session/matchmaking: Steam lobbies for party formation, invites, ready-state, and metadata; then handoff to host/dedicated connection. Support direct IP/server favorites for dedicated. Defer PlayFab/GameLift/EOS unless you later need hosted fleet allocation.
- Dedicated architecture: phase 1 “headless-ish exe” is acceptable. Phase 2 only if needed. Do not attempt seamless host migration in v1; save-and-rehost is the pragmatic path.
- Security/trust: disable the dev named-pipe/Lua backdoor in multiplayer, require mod/version manifest handshake, separate trusted private-host mode from any future secure/dedicated progression, and accept that host abuse is unsolved in host-authoritative P2P.
