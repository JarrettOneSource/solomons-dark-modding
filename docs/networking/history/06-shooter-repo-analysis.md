**2. Netcode architecture it demonstrates**
Authority is server-authoritative. The server owns one `world`, one per-client input queue, per-client ack state, and a history ring for rewind ([server/server.ts:33](</mnt/c/Users/User/Documents/GitHub/NetworkingExample/server/server.ts:33>)-[server/server.ts:47](</mnt/c/Users/User/Documents/GitHub/NetworkingExample/server/server.ts:47>)). Inputs are queued with both `seq` and `tick`, sorted, applied only when due, and old movement never rewinds the sim; if no newer input arrives, `lastAppliedInput` keeps movement continuous ([server/server.ts:107](</mnt/c/Users/User/Documents/GitHub/NetworkingExample/server/server.ts:107>)-[server/server.ts:163](</mnt/c/Users/User/Documents/GitHub/NetworkingExample/server/server.ts:163>), [server/server.ts:265](</mnt/c/Users/User/Documents/GitHub/NetworkingExample/server/server.ts:265>)-[server/server.ts:289](</mnt/c/Users/User/Documents/GitHub/NetworkingExample/server/server.ts:289>)). Shooting is a separate event path: the client sends `shoot`, the server rewinds history, traces against walls then players, applies authoritative damage, and broadcasts a shot result ([client/network.ts:224](</mnt/c/Users/User/Documents/GitHub/NetworkingExample/client/network.ts:224>)-[client/network.ts:237](</mnt/c/Users/User/Documents/GitHub/NetworkingExample/client/network.ts:237>), [server/server.ts:291](</mnt/c/Users/User/Documents/GitHub/NetworkingExample/server/server.ts:291>)-[server/server.ts:366](</mnt/c/Users/User/Documents/GitHub/NetworkingExample/server/server.ts:366>), [shared/lagcomp.ts:182](</mnt/c/Users/User/Documents/GitHub/NetworkingExample/shared/lagcomp.ts:182>)-[shared/lagcomp.ts:261](</mnt/c/Users/User/Documents/GitHub/NetworkingExample/shared/lagcomp.ts:261>)).

**4. Naive or wrong patterns for your use case**
`Reliable ordered WebSocket for everything:` Snapshot/state traffic, shot events, and input all ride the same reliable ordered stream ([client/network.ts:203](</mnt/c/Users/User/Documents/GitHub/NetworkingExample/client/network.ts:203>)-[client/network.ts:237](</mnt/c/Users/User/Documents/GitHub/NetworkingExample/client/network.ts:237>), [server/index.ts:46](</mnt/c/Users/User/Documents/GitHub/NetworkingExample/server/index.ts:46>)-[server/index.ts:55](</mnt/c/Users/User/Documents/GitHub/NetworkingExample/server/index.ts:55>), [server/index.ts:75](</mnt/c/Users/User/Documents/GitHub/NetworkingExample/server/index.ts:75>)-[server/index.ts:89](</mnt/c/Users/User/Documents/GitHub/NetworkingExample/server/index.ts:89>)). `Full-world JSON snapshots at 60 Hz:` Snapshot is just the entire `World`, deep-cloned with JSON, then sent every tick to every client ([shared/types.ts:40](</mnt/c/Users/User/Documents/GitHub/NetworkingExample/shared/types.ts:40>)-[shared/types.ts:54](</mnt/c/Users/User/Documents/GitHub/NetworkingExample/shared/types.ts:54>), [shared/snapshot.ts:8](</mnt/c/Users/User/Documents/GitHub/NetworkingExample/shared/snapshot.ts:8>)-[shared/snapshot.ts:17](</mnt/c/Users/User/Documents/GitHub/NetworkingExample/shared/snapshot.ts:17>), [server/index.ts:92](</mnt/c/Users/User/Documents/GitHub/NetworkingExample/server/index.ts:92>)-[server/index.ts:113](</mnt/c/Users/User/Documents/GitHub/NetworkingExample/server/index.ts:113>)).
