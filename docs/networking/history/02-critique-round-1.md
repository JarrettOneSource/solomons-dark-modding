REFUTE

Reason: the current repo does not actually support native remote participants or a true dedicated server on the existing rail.

The authority split is directionally fine, but the reuse premise is false: bots are not yet a drop-in remote-human path.

Do not extend the fixed packet structs; replace them with a variable-size protocol envelope plus explicit world-snapshot/resync messages.

Ship order should be: host-client P2P first, generic participant rail second, true dedicated only after a real GameServer bootstrap and a safe no-slot0 pump exist.

**Findings**
1. Native remote participants still do not exist on the current rail. Refs: `Mod Loader/SolomonDarkModLoader/include/multiplayer_runtime_state.h:81`, `Mod Loader/SolomonDarkModLoader/src/bot_runtime/helpers.inl:109`, `Mod Loader/SolomonDarkModLoader/src/mod_loader_gameplay/dispatch_and_hooks_execute_requests.inl:92`, `Mod Loader/SolomonDarkModLoader/src/mod_loader_gameplay/bot_registry_and_movement_participant_snapshot.inl:134`. The model can represent `RemoteParticipant + Native`, but the materialization path only reads `ReadBotSnapshot` and only treats `LuaBrain` participants as live. A native remote sync request is silently treated as a stale bot request. Proposed correction: add a generic `ReadParticipantSnapshot` / controller-source interface keyed by `participant_id`, keep the same `ParticipantEntityBinding`, and swap the producer from Lua to network without destroying the actor.

2. “Headless-ish dedicated exe” is not a real dedicated strategy with the current execution model. Refs: `Mod Loader/SolomonDarkModLoader/src/d3d9_end_scene_hook.cpp:27`, `Mod Loader/SolomonDarkModLoader/src/mod_loader_gameplay/dispatch_and_hooks_pump_loop.inl:28`, `Mod Loader/SolomonDarkModLoader/src/mod_loader_gameplay/dispatch_and_hooks_tick_and_render_hooks.inl:328`, `Mod Loader/SolomonDarkModLoader/src/mod_loader_gameplay/scene_and_animation_memory_and_progression.inl:152`, `Mod Loader/SolomonDarkModLoader/src/mod_loader_gameplay/dispatch_and_hooks_execute_requests.inl:267`, `Mod Loader/SolomonDarkModLoader/src/steam_bootstrap.cpp:69`, `Mod Loader/SolomonDarkModLoader/src/steam_bootstrap_api.cpp:110`. The repo already moved gameplay mutations off EndScene and onto local slot-0 `PlayerActorTick`, which is safer, but it also means the current world-mutation rail assumes a live local player actor/world. On top of that, the Steam bootstrap is client-side `SteamAPI_Init` with friends/matchmaking/networking/user interfaces only; there is no `SteamGameServer_Init`, no anonymous server login, and no server-token flow. Proposed correction: treat hidden-client hosting as a bring-up harness only; real dedicated requires a separate GameServer bootstrap or non-Steam dedicated transport, plus a safe pump that is not keyed to slot 0. Steam docs: [API overview](https://partner.steamgames.com/doc/sdk/api?l=english&source=post_page---------------------------), [ISteamGameServer](https://partner.steamgames.com/doc/api/ISteamGameServer).

3. The current protocol scaffold should be replaced, not extended, and it is already incomplete even for players. Refs: `Mod Loader/SolomonDarkModLoader/include/multiplayer_runtime_protocol.h:28`, `Mod Loader/SolomonDarkModLoader/include/multiplayer_runtime_protocol.h:35`, `Mod Loader/SolomonDarkModLoader/include/multiplayer_runtime_protocol.h:61`, `Mod Loader/SolomonDarkModLoader/src/multiplayer_foundation.cpp:17`, `Mod Loader/SolomonDarkModLoader/src/mod_loader_gameplay.cpp:525`, `Mod Loader/SolomonDarkModLoader/src/mod_loader_gameplay/dispatch_and_hooks_execute_requests.inl:42`. `StatePacket` has no `participant_id`, `steam_id`, or `scene_intent`, and the file only defines fixed-size structs. At the runtime layer, the supposed shared entity binding is still wizard/bot-shaped, and the alternate `RegisteredGameNpc` rail is explicitly disabled. Proposed correction: keep `SDMP` and versioning if you want, but replace the packet schema with a variable-size envelope and separate payload families for participants, enemies, and drops. Critical missing messages are participant join/leave/profile, enemy spawn/snapshot/event/despawn, drop spawn/pickup/despawn, cast-hit/damage confirmation, and reliable world resync. Chat is optional.

4. Do not add a parallel sim thread. Refs: `Mod Loader/SolomonDarkModLoader/src/runtime_tick_service.cpp:18`, `Mod Loader/SolomonDarkModLoader/src/bot_runtime.cpp:21`, `Mod Loader/SolomonDarkModLoader/src/mod_loader_gameplay/dispatch_and_hooks_pump_loop.inl:28`, `Mod Loader/SolomonDarkModLoader/src/mod_loader_gameplay/dispatch_and_hooks_tick_and_render_hooks.inl:514`, `Mod Loader/SolomonDarkModLoader/src/mod_loader_gameplay/bot_registry_and_movement_tick_and_movement.inl:552`, `Mod Loader/docs/multiplayer-mvp-to-beta-roadmap.md:169`. The only existing fixed threads are 50 ms utility loops; actual actor motion, collision, AI, and safe world mutation still live on the stock game thread. The repo already has documented crash history from doing gameplay work in the wrong phase. Proposed correction: keep the stock game thread authoritative, sample from safe hook points after stock updates, and run a separate replication scheduler that down-samples those samples to 20/30 Hz.

5. Enemy replication is under-specified at the seam level, and 10 Hz is an unproven lower bound. Refs: `Mod Loader/SolomonDarkModLoader/src/mod_loader_run_lifecycle.cpp:81`, `Mod Loader/SolomonDarkModLoader/src/mod_loader_run_lifecycle.cpp:334`, `Mod Loader/SolomonDarkModLoader/src/mod_loader_run_lifecycle.cpp:362`, `Mod Loader/docs/pathfinding-investigation.md:162`. Current hooks cover enemy spawn and death, but not damage, attack-start, teleport, or boss-phase/state transitions. The recovered enemy behavior is class-specific tick/state-machine code layered on shared chase helpers; I found no Banshee-specific instrumentation in the repo. Proposed correction: recover or synthesize seams for damage/state transitions before freezing the packet taxonomy, and assume 20 Hz minimum for normal enemies and 30 Hz or explicit eventization for bosses/teleports.

6. Drop replication is not ready for “drops” in the general sense; it is ready for gold orbs only. Refs: `Mod Loader/SolomonDarkModLoader/src/mod_loader_run_lifecycle.cpp:408`, `Mod Loader/SolomonDarkModLoader/src/mod_loader_gameplay/dispatch_and_hooks_execute_requests.inl:667`, `Mod Loader/docs/inventory-re-task.md:48`, `Mod Loader/docs/inventory-re-task.md:104`, `Mod Loader/docs/inventory-item-investigation.md:275`. The repo only has a verified gold drop path and no recovered walk-over pickup bridge into inventory. Proposed correction: either scope beta to gold-only drops first, or finish RE for non-gold drop factories and the pickup callback chain before promising authoritative loot sync.

7. Reconnect/join-in-progress needs an explicit reliable world snapshot. Refs: `Mod Loader/SolomonDarkModLoader/src/multiplayer_service_loop.cpp:47`, `Mod Loader/SolomonDarkModLoader/src/mod_loader_gameplay/public_api_state_getters.inl:1`, `Mod Loader/docs/multiplayer-mvp-to-beta-roadmap.md:332`. There is no session manager, no peer catalog, and no world-snapshot builder in the multiplayer layer today. Incremental snapshot bursts are not enough for a fresh or reconnecting client. Proposed correction: add a reliable chunked `WorldSnapshot` that includes participant list, run header (`run_nonce`, wave, elapsed, scene), live enemies, live drops, and each player’s progression/loadout/vitals before switching to incremental updates.

8. The Lua surface is multiplayer-fatal unless you gate it hard. Refs: `Mod Loader/SolomonDarkModLoader/src/lua_engine_bindings.cpp:134`, `Mod Loader/SolomonDarkModLoader/src/lua_engine_bindings_gameplay.cpp:309`, `Mod Loader/SolomonDarkModLoader/src/lua_engine_bindings_input.cpp:102`, `Mod Loader/SolomonDarkModLoader/src/lua_engine_bindings_bots.cpp:6`, `Mod Loader/SolomonDarkModLoader/src/lua_engine_bindings_debug.cpp:41`, `Mod Loader/SolomonDarkModLoader/src/lua_engine_bindings_debug/functions_typed_readers_and_writers.inl:61`, `Mod Loader/SolomonDarkModLoader/src/lua_engine_bindings_debug/functions_field_accessors.inl:19`, `Mod Loader/SolomonDarkModLoader/src/lua_engine_bindings_debug/functions_native_calls.inl:1`, `Mod Loader/SolomonDarkModLoader/src/lua_exec_pipe.cpp:105`, `Mod Loader/SolomonDarkModLoader/src/mod_loader.cpp:327`. Every Lua mod gets `sd.debug`, and enabling the Lua engine always starts the named-pipe exec server. Public APIs already mutate gameplay (`sd.input.*`, `sd.hub.start_testrun`, `sd.gameplay.start_waves`, `sd.world.spawn_*`, `sd.bots.*`), while debug APIs expose arbitrary memory writes and arbitrary native calls. Proposed correction: in multiplayer builds, disable the exec pipe, do not register `sd.debug`, route all net-visible mutations through host intent RPCs, and leave only read-only APIs/UI snapshot locally accessible.

**Next Actions**
1. Stop treating bots as proof of remote-human readiness. Implement a generic participant snapshot/controller source keyed by `participant_id`, remove the Lua-only filters, and prove one live `LuaBrain -> Native` swap without respawning the actor.
2. Replace `multiplayer_runtime_protocol.h` with a real message schema: common envelope, separate participant/enemy/drop payload families, reliable world snapshot/resync, and host-confirmed event packets.
3. Cut true dedicated out of the first multiplayer milestone unless you first add a real server bootstrap (`SteamGameServer` or non-Steam dedicated transport), a safe no-slot0 game-thread pump, and hard multiplayer gating for Lua debug/exec features.
