# Native enemies, bosses, helpers, and drops

## Result and evidence boundary

The retail executable has 19 runtime types that participate in the enemy
system: the `Badguy` base, thirteen stock hostile actor families, one allied
Imp variant, two subordinate attack/containment actors, one helper, and the
Imp Portal spawner. This is a finite compiled census, not a directory scan.
Every type is allocated by `GameObjectFactory (0x005B7080)` and installs a
fixed vtable.

The machine-readable result is
[`native-enemy-catalog.json`](native-enemy-catalog.json), generated by
`tools/build_native_enemy_catalog.py`. It joins the finite type list to the
native object catalog and preserves allocation size, constructor, vtable,
selected lifecycle slots, config controls, spawned child types, drop policy,
behavior evidence, and every directly attributable atlas range/consumer.

The static family pass is complete. Two deliberately narrow runtime checks
remain in the later live-validation phase:

- Dire Faculty's CFG/UI exposes primary index `3` (`FAUST JET`) and secondary
  index `3` (`THING OF ICE`), but `0x004804D0` has no event branch for either.
  The static result is **branch absent**, not a guessed spell implementation.
- Portal frequency selects randomized countdown bounds, but Ghidra did not
  recover the enum switch arithmetic cleanly enough to publish exact bounds.
  The ownership and spawn/reset lifecycle is statically proved.

## Shared construction and combat lifecycle

The enemy pipeline is:

```text
enemy CFG text
    |
    v
MonsterSetup_Parse (0x004AFBC0)
    |
    +--> native defaults (0x00640240)
    +--> wave flag text -> internal codes (0x0062E070)
    |
    v
BuildEnemyConfig (0x0046B390)
    |
    +--> arena difficulty/scenario scalars at +0x9008..+0x9024
    |
    v
factory allocation (0x005B7080 / 0x00463B50)
    |
    v
apply family config (0x00462790)
    |
    v
place/register actor (0x00469580)
    |
    +--> target refresh (0x00483480)
    +--> shared chase/pathfinding (0x004835F0)
    +--> family tick/action/render slots
    +--> base contact (0x0048A290)
    |
    v
death/reward presentation (0x004819D0 / family override)
    |
    v
drop selector (0x0047C070)
```

Target identities use the engine's kind/index handle pair rather than a raw
long-lived pointer. Target refresh resolves candidate actors, applies team and
validity checks, and stores the selected identity; `0x00484B30` and
`0x00482E90` clear targets/release callbacks when either side disappears.
Concrete family ticks can call the shared chase loop or replace it with an
orbit, boss action, stationary-spawner, or subordinate-child state machine.
The pathfinding/flanking behavior and its actor fields are cross-checked in
[pathfinding-investigation.md](../pathfinding-investigation.md).

payload that multiplayer must agree on.

## Wave flag grammar and transformations

`0x0062E070` maps text tokens to internal codes; `0x0046B390` applies those
codes. The full structured mapping is in the generated catalog. The important
compiled transformations are:

| Tokens | Internal codes | Build effect |
| --- | --- | --- |
| `HPUP`, `HPDOWN` | `0x01`, `0x02` | HP x1.5 / x0.5 |
| `STRONG`, `WEAK` | `0x03`, `0x04` | damage fields x1.5 / x0.5 |
| `FAST`, `SLOW` | `0x05`, `0x06` | native fast multiplier / chase and attack x0.5 |
| `XPBONUS`, `BURNING` | `0x07`, `0x08` | XP bonus; flaming plus chase/attack x1.5 |
| `HELM`, `HORNED`, `HOODED` | `0x09..0x0B` | headgear 1..3 plus distinct flat HP additions |
| `LEADING`, `SCATTERSHOT`, `RANDOMSHOT` | `0x0C`, `0x0D`, `0x31` | archer accuracy/multi-shot modes 1..3 |
| `RANGEUP`, `RANGEDOWN`, `RANGEEASY` | `0x0E..0x10` | range selectors 2, 1, 3 |
| `SHIELD`, `SHIELDOTHERS` | `0x11`, `0x12` | self/other shield, interval 10, strength 50 |
| `SHIELDSTRONG`, `SHIELDFAST` | `0x13`, `0x14` | shield strength x9 / interval halved |
| `SPLIT`, `SPLITMANY` | `0x15`, `0x16` | random 1..2 / wave-scaled split count |
| `MANYMAGGOTS`, `STRONGMAGGOTS` | `0x17`, `0x18` | max Maggots 50 / Maggot HP and damage 5 |
| `POISONARROW`, `FIREARROW` | `0x19`, `0x1A` | arrow type 2 with derived extra damage / arrow type 1 |
| `ARMOR` | `0x1B` | enable armor and adjust HP |
| `SWORD`, `MACE`, `FLAIL`, `AXE`, `PIKE` | `0x1C..0x20` | weapon selector plus weapon-specific HP/damage adjustment |
| `CASTFIRE`, `CASTLIGHTNING`, `CASTFROST`, `CASTPOISON` | `0x21..0x24` | mage element 0..3 and element-specific damage scale |
| `ROTTEN` | `0x25` | flyblown plus derived poison damage/duration fields |
| `DEATHIMPS`, `DEATHIMPSMANY` | `0x27`, `0x28` | death-child count 5 / 15 |
| `ARMORMAYBE` | `0x2A` | random armor byte |
| `NOSKELETONS` | `0x2B` | multiply config `+0x60` by the native reduction constant |
| `MORESKELETONS` | `0x2C` | clear config byte `+0x94` |

Three parser/build anomalies matter to tools that reproduce the grammar:

1. an unreachable second `FLAG_ARMOR` comparison returns `0x29`, after the
   earlier `FLAG_ARMOR` branch already returned `0x1B`;
2. `FLAG_ARMORMAYBE` returns `0x2A`, not `0x29` (an older pseudo-source note
   had this wrong);
3. the builder handles internal `0x2D..0x30`—increment/decrement `+0x84`, an
   XP penalty, and setting `+0x95`—but the recovered text parser emits none of
   those codes.

Code `0x26` is also inserted internally for one Coffin construction variant;
it writes `25` to config `+0x90`. It is not a parsed text flag.

## Rewards and ground-drop selection

Death presentation and reward accounting feed `0x0047C070`. Maggot and Cocoon
return before candidate construction, so they can never produce normal death
loot. GoodImp and Crow do not enter the hostile reward path.

The selector seeds its deterministic RNG from actor `+0x1C0`. It evaluates
the six CFG selectors independently, applies the arena disable mask at
`arena +0x8F04`, builds an eligible candidate list, and chooses **exactly one**
candidate. Multiple true selector rolls do not materialize multiple drop
categories.

| Arena mask | Suppressed candidate |
| ---: | --- |
| bit `0` | gold |
| bit `1` | potion |
| bit `2` | power-up |
| bit `3` | orb |
| bit `4` | key |
| bit `5` | item |

Materialization is category-specific:

| Candidate | Native path | Created type/behavior |
| --- | ---: | --- |
| Key | `0x00468440` | `Item_Misc 0x1B64` |
| Orb | factory | `Orb 0x7DB` |
| Gold | `0x0046AA90` | `Gold 0x7DC`; chunks at 25 and participates in sack merging |
| Item | `0x0046A360` | selected item definition/object |
| Power-up | factory | `Bonus 0x7F6` |
| Potion | `0x0046AE20` | `Item_Potion 0x1B59` |

A separate quick-potion path runs before normal candidate construction.
Gold selector value `5` also reaches an extra gold-materialization branch.
The later ground-loot pass documents object lifetime, merge, render, pickup,
and inventory insertion; this section proves enemy-side eligibility and
selection only.

## Skeleton family

### Skeleton

Skeleton combines a shared locomotion/state skeleton with independently
selected weapon, headgear, armor, and burning presentation. Weapon values map
to unarmed, sword, mace, flail, axe, and pike behaviors; the wave builder also
changes HP/damage per weapon instead of treating the value as cosmetic.
Armor and headgear likewise affect the built config before actor creation.

`Skeleton_Tick (0x00484B90)` selects movement/attack state. The articulated
renderer `0x0048DEE0` composes body/equipment ranges rather than selecting one
monolithic sprite. `0x0048D2A0` runs equipment-aware death presentation;
`0x00477020` is the shared special-death path. Flaming Skeletons own a
`Fire 0x7E3` effect whose lifecycle must be cleaned up with the actor. The
bone/gear disassembly details and fragment semantics are cross-checked in
[skeleton-death-effects-re.md](../skeleton-death-effects-re.md).

### Skeleton Archer

`0x00485200` maintains distance and optional strafing. `0x00477B90` creates
`Arrow 0x7DA`, transfers team/source/damage, selects normal/fire/poison arrow
payload, and applies accuracy mode plus extra-arrow count. `LEADING`,
`SCATTERSHOT`, and `RANDOMSHOT` are distinct mode values; extra arrows are a
separate count, so they must not be collapsed into one “multishot” flag.

### Skeleton Mage

`0x00490860` owns cast timing and shield cadence; `0x0047FDE0` dispatches the
selected element. Proven spawned objects include `Firebolt 0x7EB` and
`GuidedMissile 0x7EC`; other element branches use direct/projectile and status
paths documented in the projectile catalog. Self shield and ally shield have
separate toggles and strengths but share the configured interval. Headgear,
cloak, range, element, shielding, and flaming are all independent config
axes.

## Imp variants

`Imp 0x3EC` uses `0x00485DC0` for its flying chase and `0x00492E10` for
rendering. `0x004824A0` handles death splitting. `SPLIT` chooses one or two
children; `SPLITMANY` derives a count from wave progression. Child Imps are
created through the config path rather than cloned as raw memory, which lets
the path reduce/retain split state and makes recursive splitting possible.

`GoodImp 0x3ED` is a different factory class and team relationship, not a
green recolor of hostile Imp. It targets hostile actors, expires on a
temporary ally lifetime, produces final Fire presentation, and does not run
hostile rewards/drops.

`GreenImp 0x7FC` shares the chase logic but renders from Unholy records. It is
created by `UnholySpit_Impact (0x005EA6F0)` only when the projectile's
`SPIT IMPS` payload byte is set and the sampled placement test accepts that
point.

## Zombie and Wraith

Zombie contact creates two Knockback modifier paths and a `Poisoned 0x1B72`
modifier when configured poison damage is positive. Flyblown and body-type
selectors control presentation, while poison punch DPS, PoisonPool DPS, and
duration remain separate numeric fields. Death creates `PoisonPool 0x806`
with the configured pool payload before completing common rewards/drops.

Wraith does not merely run straight chase. `0x00486C30` alternates approach,
orbit/retreat, attack, and fade/visibility state while maintaining target
identity. Its hit path applies `Dazzle 0x1B6E`; flaming adds the shared burn
presentation. Death art/reward presentation uses `0x00495600`.

## Bosses

### The Discorporeal (`DemonSkull`)

Config bytes `+0x94..+0x97` become a live capability bitfield:

| Bit | Config option | Runtime action |
| ---: | --- | --- |
| `1` | Eye Lasers | state `0x1A`; creates two `EyeLaser 0x7FF` objects from the two eye sockets |
| `2` | Mouth Beam | state `0x1B`; `0x0044FFE0` scans beam geometry and applies configured beam DPS |
| `4` | Spit Fire | state `0x1C`; `0x00449A00` creates `UnholySpit 0x7FB` |
| `8` | Spit Imps | copied to Unholy Spit; augments its impact payload |

`0x00474930` stores the scheduled selector and chooses the animation state.
Bite/lunge is state `0x18`; scream and flair occupy separate later action
states and are not config bit 8.

Unholy Spit travels using origin, velocity, lifetime, and fire-damage fields.
At detonation `0x005EA6F0` always places `GreenFire 0x7FA` along the impact
trajectory. With the copied `SPIT IMPS` byte, it also samples the trajectory,
runs the arena placement predicate, and creates `GreenImp 0x7FC` at accepted
points. Each child inherits team and receives scaled primary/secondary damage
from the projectile. This is why treating “Spit Imps” as a fourth scheduler
entry produces the wrong behavior.

### Lesser Demon

The Demon renderer `0x00498BA0` composes six articulated groups from the
Demon atlas. Event `0x0049A270` creates `DemonBomb 0x7F7` and transfers primary
damage. Death `0x00482930` first performs common boss presentation/rewards,
then creates the configured number of radial `Imp 0x3EC` children with the
boss arena/team/combat scalars. The boss tick can also emit Fire presentation.

### Dire Faculty

Config stores primary choice at live `+0x24C`, secondary at `+0x250`, hat at
`+0x23C`, gender at `+0x244`, and the two damage values in the normal combat
fields. `0x00475340` usually schedules a primary action but can select the
secondary state when its cooldown/random condition permits.

| Primary index | UI name | Recovered event behavior |
| ---: | --- | --- |
| `0` | Death Missile | state `0x1F`; `SkullMissile 0x800` with primary damage |
| `1` | Blightning | state/action `0x21`; direct geometric lightning scan with primary damage |
| `2` | Direball | state `0x1F`; `DarkFireball 0x804` with primary damage |
| `3` | Faust Jet | string/config value exists; no `0x004804D0` branch observed |

| Secondary index | UI name | Recovered event behavior |
| ---: | --- | --- |
| `0` | Acid Pain | `RainOfBones 0x801` |
| `1` | Tragic Circle | `TragicCircle 0x802` |
| `2` | Ring of Dire | radial/multiple `DarkFireball 0x804` objects |
| `3` | Thing of Ice | string/config value exists; no `0x004804D0` branch observed |

The missing index-3 cases are a stock executable finding. A mod loader should
not advertise them as functional merely because their labels parse.

### Heartmonger and Crow ownership

`0x00488F20` allocates exactly the configured number of 0x6C-byte Crow
helpers and gives them evenly spaced orbit phases. Heartmonger owns their
container and reattaches/updates them each tick. Blind-chance enum values are
`0%, 5%, 10%, 30%, 100%`; the resulting percentage is stored at live
`+0x2A4`.

Crow `0x0047EA70` stores parent identity at helper `+0x14` and target identity
at `+0x68`. While the parent cooldown permits, it periodically scans actors in
its distance/angle window, selects a target, dives, and returns to orbit.
`0x0047A160` applies Heartmonger primary damage. It attempts blind only when
the struck actor is the player and `RandomInt(100)` is below the configured
percentage.

SKELEFRIENDS chooses a randomized interval/countdown at Heartmonger
`+0x2B0/+0x2AC`. On expiry `0x00489000` can build and spawn
`SkeletonArcher 0x3EA`, `Skeleton 0x3E9`, or one/two `Imp 0x3EC` actors with
selected modifier arrays. Death `0x0049FB60` clears parent ownership and
registers surviving Crow helpers independently before creating the boss
fragment field and completing rewards/drops.

## Coffin and Maggot ownership

Coffin applies max Maggots to live `+0x22C`, child HP to `+0x230`, child
primary damage to `+0x234`, and child poison damage to `+0x238`.
`Coffin_Tick (0x004A2760)` is a four-state machine:

1. closed/waiting;
2. opening/charging;
3. transition delay;
4. open/active.

Opening invokes `0x00479C30` three times. While open, Coffin can replenish
children according to charge/health state and its live-child count. The spawn
helper creates `Maggot 0x7FD` on one of two lid/edge launch trajectories,
copies arena/team identity and configured combat values, writes the Coffin
identity into the child, and registers the child in the Coffin container.
The open state also scans nearby actors for its poison/damage logic; Golem is
excluded from poison.

Maggot first runs ballistic emergence `0x0047E410`, then crawling logic. Its
periodic parent check resolves the stored kind/index pair. If the parent is
gone or invalid, cleanup removes the child; when the parent still resolves,
the appropriate cleanup/bite path updates Coffin's live count. A valid nearby
target receives primary damage and, when positive, a Poisoned modifier except
for Golem. The Maggot then invokes its own death immediately: a bite is
single-use. `0x0047C070` explicitly suppresses Maggot drops.

## Spider, Silk, and Cocoon

Spider applies cocoon HP to live `+0x24C`, suck DPS to `+0x248`, and the
spit-web toggle to `+0x250`. `0x0047A580` owns locomotion/action states `0..5`.
`0x00475CD0` begins hop/lunge. `0x00475DB0` permits web spit only when enabled
and a valid target exists.

`0x00475AC0` does **not** create Cocoon. It changes Spider state, creates
`Silk 0x808`, copies position/aim/team and cocoon-strength payload, then sets
the attack cooldown. Silk has its own projectile tick (`0x005F8B50`), render
path (`0x0060F590`), and trail renderer (`0x005F92C0`).

On actor collision, Silk creates `Mod_Webbed 0x1B79` (`0x00623B10`), copies
its cocoon-strength payload from Silk `+0x22C` to modifier `+0x20`, and applies
the modifier to the struck target. `0x00623B50` marks target flag
`+0x138 bit 0x20` and scales target movement field `+0x218` by the web
severity. Repeated applications merge in `0x00627BD0`: severity `+0x1C`
increases, payload `+0x20` keeps the maximum value, and the first transition
to the compiled full-web threshold calls `0x0052C680` with the target actor as
`this`.

The duplicate-safe Cocoon handle is therefore target-owned:

1. keep the maximum web/cocoon payload at target `+0x20C`;
2. resolve the Cocoon identity stored at target `+0x214/+0x216`;
3. if it still resolves, return without creating another Cocoon;
4. otherwise allocate `Cocoon 0x80A` at the target position, write the target
   identity to Cocoon `+0x210/+0x212`, register it, and store the new Cocoon
   identity back on the target.

Cocoon initialization snapshots its fixed position at `+0x214/+0x218`;
`0x00475D10` keeps the Cocoon there while full web severity immobilizes the
target. `0x0048BCE0` resolves the target identity, handles release/death
presentation, clears callbacks when the target is gone, and then reaches base
contact/death behavior. Cocoon drops are explicitly suppressed.

Spider's grab/suck state applies primary damage on initial contact, follows
the held target while it remains valid, and every 25 ticks applies half of
configured suck DPS. Contact can also transition into the jump state before
calling base contact. Death uses the normal reward/drop path.

## Imp Portal

Portal spends its first ten ticks materializing, then remains stationary.
Live `+0x230` is the spawn countdown; the config enum selects randomized
bounds stored at `+0x234/+0x238`. At zero, `0x00489CC0` creates
`Imp 0x3EC`, places it at the Portal, marks it airborne/ejected, copies team
and primary damage, registers it, and resets the randomized countdown.
Contact `0x0048C370` adds impact particles and then calls base contact. Death
`0x004A1FA0` performs common rewards/drop selection and a large portal
fragment/decal presentation.

## Custom-content and multiplayer consequences

The native enemy seam has four distinct levels:

1. **CFG replacement/tuning:** existing family fields, flags, drop selectors,
   and numeric values can change behavior already implemented by that class.
2. **Atlas replacement:** art can replace fixed records/ranges while retaining
   the native record ABI and resident-page lifetime.
3. **Lua/native hooks:** new behavior can intercept construction/tick/action
   paths or create existing factory types, but a new CFG token alone cannot
   add a vtable, action dispatcher branch, projectile class, or factory ID.
4. **New compiled/native-backed type:** requires extending the factory/type
   registry and every serialization/network consumer that assumes the finite
   type set. This is not provided by stock data loading.

For deterministic hosted play, peers must agree before gameplay construction
on enemy CFG bytes/evaluated values, wave flag arrays, arena difficulty
scalars, factory/type semantics, child-spawn behavior, drop-disable masks, and
any Lua/native hooks. Art that changes only pixels is presentation state, but
bundle geometry/record layout must still match the consumer ABI. Behavior,
collision geometry, timing, RNG call order, or record-count changes are parity
state and cannot be treated as a cosmetic-only download.

No website/mod-download code was changed during this pass.
