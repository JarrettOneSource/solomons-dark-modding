# Native asset system

## Scope and build identity

This document describes how the retail `SolomonDark.exe` loads, decodes,
uploads, stores, restores, and destroys visual assets. It is based on static
analysis of the following executable and on a byte-for-byte inventory of the
shipped `images` directory:

| Property | Value |
| --- | --- |
| Executable | `SolomonDarkAbandonware/SolomonDark.exe` |
| SHA-256 | `03a834566ce70fd8088f4cf9ee6693157130d8aec28c092cb814d6221231f1e3` |
| Architecture | 32-bit x86 PE |
| Sprite records | 10,498 across 28 bundles: 9,780 direct common records plus 718 font-wrapper glyph records |
| Loose image names | 12 |

Addresses in this document are image-base virtual addresses for that exact
executable. Names beginning with `Native_` are descriptive names assigned by
this investigation, not original debug symbols.

Evidence labels used below:

- **Confirmed**: direct decompilation/disassembly or exhaustive file parsing.
- **Correlated**: static code and extracted pixels/configuration agree, but a
  final runtime observation would improve the semantic name.
- **Inferred**: the structure is known but its human-facing meaning is still
  provisional.

paintbkg.png      rise.png         river.png       road.png

The deterministic machine-readable inventory is
[`native-content-inventory.json`](native-content-inventory.json), generated by
`tools/build_native_content_inventory.py`. It records every current file's
relative path, size, SHA-256, content class, every atlas dimension/count,
auxiliary font groups, and all recovered builder destinations. Every auxiliary
group includes its exact glyph IDs, kerning triples, serialized record range,
and owning runtime-object destination. The audited
tree currently has 507 files: 496 installed-content files and 11 files under
the runtime-mutable `sandbox/` tree. The latter includes user boneyards,
portrait/save/cache state, and settings and must not be treated as baseline
distributable content merely because it is beneath the game directory.

Image lookup is extension-flexible even though the retail copy uses PNGs. The
native probe order is `.cached`, `.png`, `.thumb`, `.gif`, `.jpg`, and `.raw`.
An optional sibling whose base name ends in `_alpha` supplies a separate alpha
channel. See [Image lookup and decoding](#image-lookup-and-decoding).

## Bundle construction pipeline

Each atlas has a generated-looking build function. The common sequence is:

1. Reset atlas-dimension override hints with `0x00413020`.
2. Ask `0x004136F0` whether to use an `@2X` variant. In this retail binary
   that function always returns false, so every stock builder selects its base
   name; the alternate branch is compiled but dormant.
3. Acquire image pages with `Native_SpriteBundle_LoadImagePageSet`
   (`0x00413030`).
4. Open `images\\<name>.bundle` with
   `Native_SpriteBundle_OpenMetadataStream` (`0x004134B0`).
5. Read a fixed, compiled-in number of descriptors with
   `Native_SpriteBundle_ReadRecord` (`0x00413B10`).
6. Register/finalize the page handles with `0x0041FFE0`.
7. Close only the metadata stream and its temporary buffer with `0x00413850`.
   The resident pages remain owned by the bundle reference count.

This is not a name-addressed sprite database. Record order is an ABI: each
builder reads descriptor zero into a specific object field, then descriptor
one into the next compiled destination, and so on. Most repeated animation
groups are dynamic arrays of 0xC4-byte runtime sprites. Replacing a bundle with
records in a different order changes which art every native consumer sees.

## Serialized sprite descriptor

The common record is little-endian and has no file header:

```text
record_size = 45 + 8 * point_count
```

| Offset | Type | Native destination | Meaning |
| ---: | --- | --- | --- |
| `0x00` | `f32` | temporary | Atlas X |
| `0x04` | `f32` | temporary | Atlas Y |
| `0x08` | `f32` | temporary | Packed width |
| `0x0C` | `f32` | temporary | Packed height |
| `0x10` | `i32` | sprite `+0x94` | Logical/untrimmed width |
| `0x14` | `u32` | sprite `+0x98` | Logical/untrimmed height |
| `0x18` | `f32` | geometry inputs | Trimmed content width |
| `0x1C` | `f32` | geometry inputs | Trimmed content height |
| `0x20` | `f32` | geometry inputs | Content-center offset X |
| `0x24` | `f32` | geometry inputs | Content-center offset Y |
| `0x28` | `u8` | construction branch | Atlas-rotation flag |
| `0x29` | `u32` | sprite vector count | Extra point count |
| `0x2D` | `N * {f32,f32}` | sprite `+0xA4` | Extra point vector |

`Native_SpriteBundle_ReadRecord` reads this stream and calls the runtime-sprite
constructor at `0x00413DE0`. That constructor resolves rotated packing once,
builds its four local vertices at `+0x2C..+0x48`, UVs at `+0x4C..+0x68`, and
bounds at `+0x6C..+0xA0`, and stores atlas-scale factors at `+0xB4..+0xC0`.
Consequently, downstream render code consumes ready-to-draw geometry and does
not parse `.bundle` bytes again.

All 10,498 retail records were parsed exhaustively: 9,780 are direct calls to
the common record reader from generated builders and 718 are common records
nested inside the font-wrapper grammar. All rectangles are finite,
positive, integral, and within their sibling atlas; all rotation bytes are
zero. `Fonts.bundle` contains one direct sprite followed by nine font-wrapper
objects containing 626 glyph records. `ControlPanel.bundle` contains 24 direct
sprites followed by one wrapper containing 92 glyph records. The wrappers
carry a three-float header, kerning pairs, glyph IDs, glyph metrics, and the
embedded common records. Their exact runtime destinations and record ranges
are in `native-content-inventory.json`; the byte grammar and validation details
are in
[`tools/sprite-bundle-format.md`](../../tools/sprite-bundle-format.md).

## Page-set acquisition and ownership

`Native_SpriteBundle_LoadImagePageSet` (`0x00413030`) acquires the page-level
load state used by every generated atlas builder. The common base constructor
at `0x00412DF0` initializes this state; each concrete atlas is a singleton
derived object whose one-method vtable points at its generated builder.

| Bundle object offset | Meaning | Evidence |
| ---: | --- | --- |
| `+0x1C` | bundle acquisition/reference count | Incremented first; calls after the first acquire no duplicate pages. |
| `+0x24` | page-handle vector pointer | Receives the base page and any numbered siblings. |
| `+0x28` | page-handle vector capacity/end state | Used by vector growth. |
| `+0x30` | metadata I/O buffer | Lazily allocated by `0x004134B0`, freed by `0x00413850`. |
| `+0x34` | successful/resident acquisition flag | Set from the builder's success flag after pages load; returned to a waiter and reset on final release. |
| `+0x35` | threaded-build marker | Set with `+0x36` by the retail worker wrapper; exact wider coordination semantics are not named. |
| `+0x36` | threaded build in progress | Release waits for it to clear; one loader branch also waits when it is set while `+0x35` is clear. |

The first caller probes `images\\<name>` and then
`images\\<name>-1`, `images\\<name>-2`, and so forth until a page is absent.
A concurrent caller waits in 25 ms intervals while `+0x36` is set. This means
the engine already supports a logical atlas spanning numbered texture files,
but native sprite destinations and descriptor count remain fixed in code.
The generated retail builders pass a true success/residency flag and use only
the base PNG pages; no shipped atlas has numbered siblings or an `@2X` page.

`0x00413490` can launch a bundle build with `_beginthread`. Its worker at
`0x00413450` sets the 16-bit value at `+0x35` to `0x0101` (both bytes true),
invokes the bundle's virtual builder, and clears both bytes afterward. Demon,
Demon Skull, Dire Faculty, Heartmonger, and one UI path use this asynchronous
entry. `0x00413760` waits in 100 ms intervals for `+0x36` to clear before
releasing pages. The loader also contains a 25 ms wait for the state
`+0x36 != 0 && +0x35 == 0`; the worker wrapper itself sets both bytes together,
so that narrower state is documented as code behavior rather than assigned an
unsupported semantic name.

`0x0041FFE0` inserts finalized page handles into the owning renderer's global
page table at `+0x2B4`. `0x0043A6B0` grows arrays of 0xC4-byte runtime sprites;
`0x00427000` is the common four-byte-vector growth helper used for handles and
other pointer/index arrays.

`Native_SpriteBundle_Release` (`0x00413760`) is the matching release path. It
waits for asynchronous loading at `+0x36` to finish, decrements `+0x1C`, and,
when the count reaches zero, releases every page handle through `0x00420760`,
clears/frees the page vector, resets `+0x34`, closes any open metadata stream,
and discards temporary pointer-list state. Passing its force flag first sets
the count to one, guaranteeing a complete release. Concrete actor and UI
destructors call this function to balance their atlas acquisitions.

The SpriteBundle base destructor at `0x00412F70` invokes that release path and
then destroys the base arrays. This gives the resident asset three distinct
levels of ownership:

1. each screen/actor acquires the atlas singleton;
2. the singleton keeps page handles resident while its count is nonzero;
3. the renderer's global page table maps each live handle to its GPU slot.

The post-build helper `0x00413850` is narrower: it closes the `.bundle`
metadata stream after descriptors have been constructed. It intentionally
does not decrement the bundle count or unload image pages.

## Image lookup and decoding

`Native_ImageAsset_ExistsWithFallbackExtensions` (`0x0043DAD0`) searches a
base path in this order:

1. `.cached`
2. `.png`
3. `.thumb`
4. `.gif`
5. `.jpg`
6. `.raw`

The function also probes an `_alpha` companion through a guarded recursive
lookup. `Native_ImagePage_DecodeWithOptionalAlpha` (`0x0043DE70`) dispatches to
the cached, PNG, 24-bit image, GIF, or raw decoder. When an alpha companion is
present, its first component is copied into every fourth byte of the decoded
main BGRA buffer.

The `.cached` decoder starts at `0x0043EF30` and delegates unpacking to
`0x0043F010`. Its header contains a mode, width, and height:

- mode 0 is direct `width * height * 4` pixel data;
- mode 1 stores compressed color and alpha streams separately and reconstructs
  BGRA pixels.

The existence of JPEG/thumbnail/raw fallbacks is native capability, not proof
that every decoder preserves the same alpha or color fidelity. Retail atlases
use PNG, while the portrait capture path writes `.raw` files.

## GPU upload and texture table

`0x00420140` is the shared page loader. Renderer `+0x2CC` is the value of the
`Graphics.SaveVideoMemory` setting, loaded at `0x0041CE20`. When enabled, the
loader first constructs filename variants prefixed with `4444-` and `565-`
(after the final path separator). An existing `4444-` variant is decoded and
uploaded through `0x004206A0` in mode 2; an existing `565-` variant uses
`0x00420640` in mode 1. If neither exists, it falls back to the ordinary source
and 32-bit mode 0. These are optional preconverted image variants, not the
proprietary `.cached` decoder. Upload and shared table mutation are protected
by the critical section at `0x00B3F9DC`.

When an `_alpha` companion exists, `0x0043DE70` merges it into the decoded
32-bit buffer and `0x00420140` still passes upload mode 0. Instruction-level
inspection at calls `0x004204F3` and `0x004205BD` confirms both ordinary
fallback branches push mode zero, so the merged alpha byte is preserved by
this path.

`0x00441180` allocates the first free 0x20-byte texture slot in the table at
`0x00818C84`; its count is at `0x00818C88`, and `0x00443100` grows the table.
The observed Direct3D formats are `0x15`, `0x17`, and `0x1A`.
`0x00440F70` performs lock/copy/unlock and records dimensions. The upload modes
correspond to 32-bit color, an RGB565-like conversion, and A4R4G4B4.
`0x00420840` returns dimensions for a page handle. `0x004417F0` binds that
slot's Direct3D texture to texture stage zero; it is not a release function.
`0x00420760` performs page-handle release under the shared critical section,
calls `0x0043FA80` to release the slot's Direct3D texture and associated
surfaces, clears all 0x20 slot fields, unbinds the handle if it was current,
and removes it from the renderer page table.

## Device loss, restoration, and destruction

Texture slots distinguish ordinary loaded pages from persistent/recreatable
surfaces:

| Slot offset | Meaning |
| ---: | --- |
| `+0x0C` | persistent/recreate-on-device-reset flag |
| `+0x0D` | alpha-capable format flag |
| `+0x14` | secondary/scaled surface exists |
| `+0x1C` | secondary scale factor (`2` or `4`) |

`0x00441420` creates a persistent render-target slot and sets those fields.
During restoration, `0x00440520` walks persistent slots and calls `0x00441330`
to recreate the Direct3D texture, selecting format `0x15` or `0x17` from the
alpha flag and rebuilding a secondary surface when requested. It then restores
renderer-owned surfaces. `0x00440450` is the corresponding device teardown: it
releases managed slot textures, secondary surfaces, and renderer resources.

Runtime sprite destruction is separate again. A sprite is 0xC4 bytes;
`0x004138A0` initializes it, `0x00413990` releases its auxiliary arrays and
point vector, and `0x0043A840` is the scalar/vector deleting destructor.
Atlas-derived destructors destroy their inline sprites and `Array<Sprite>`
groups before invoking the SpriteBundle base destructor. That distinction
matters for custom assets: sprite descriptors, a bundle's reference count,
renderer page handles, GPU texture slots, and Direct3D device-reset state are
five different ownership layers.

## Native constraints relevant to later mod support

These are architectural facts, not yet an implementation plan:

- Native atlas lookup is path/name based, but native sprite lookup is compiled
  positional layout. A replacement atlas can safely preserve the exact bundle
  grammar and record order; adding arbitrary new native records cannot make
  stock code address them.
- A logical atlas can use multiple numbered texture pages. The `.bundle` still
  needs compatible per-record page selection and fixed native destinations.
- PNG is not the only accepted source format, and a separate `_alpha` image is
  native. A mod manifest must not assume that extension alone identifies the
  runtime asset.
- Decode buffers, bundle objects, global page handles, and GPU slots have
  distinct lifetimes. Hot replacement must account for all four; overwriting a
  file while a page is resident does not update already-built runtime sprites.
- `@2X` selection and persistent device-reset resources are native branches
  that a replacement path must preserve or explicitly supersede.
- Fonts are not plain sprite streams. Treating every `.bundle` as a flat array
  corrupts `Fonts` and the font section of `ControlPanel`.

No mod-loader changes are being made from these observations until all native
art consumers and the surrounding gameplay systems have been mapped.
